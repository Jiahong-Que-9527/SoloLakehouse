# SoloLakehouse 项目现状全方位总览

> 快照日期：2026-05-05
> 当前基线：v2.5（单轨道编排化平台）
> 文档定位：用一份文档让读者在 15 分钟内建立对项目当前真实状态的完整认知，便于后续讨论 v3 演进方向与企业级落地路径。

---

## 1. 项目定位（What it actually is）

SoloLakehouse 是一个 **可运行、可阅读、云中立** 的 Lakehouse 参考实现。

它不是：
- 一个框架或库
- 一个面向用户的 SaaS 产品
- Databricks/Snowflake 的"开源克隆"

它是：
- 一份在 **单台 Docker Compose 主机** 上完整跑通的 Lakehouse 架构样板
- 用 **纯开源组件** 演示 Databricks/Snowflake 内部是如何运转的
- 一个 **以平台工程能力为陈述对象** 的工程作品（可被 Fork、批判、复用）
- 围绕一个真实金融领域示例（ECB 利率 + DAX 指数）演示端到端 Bronze → Silver → Gold → ML 的能力

仓库当前主分支：`main`；项目语言：Python 3.13+；许可证：MIT。

## 2. 当前技术栈（v2.5 实际运行的组件）

| 层 | 组件 | 版本 | 角色 |
|---|------|------|------|
| 对象存储 | MinIO（S3 兼容） | RELEASE.2025-09-07 | Bronze/Silver/Gold Parquet + MLflow artifacts |
| 元数据库 | PostgreSQL | 17 | Hive Metastore、MLflow、Dagster、Superset 共用后端 |
| 表目录 | Apache Hive Metastore（standalone） | 4.0.0 | 表 schema/分区/位置元数据 |
| 查询引擎 | Trino | 480 | Hive + Iceberg 双 catalog |
| Gold 表格式 | Apache Iceberg（经 Trino） | — | Gold 层开放表格式 |
| 数据目录 | OpenMetadata | 1.5.x | 资产元数据 + 血缘 UI |
| BI / SQL UI | Apache Superset | 6.0.0 | 仪表盘 + 即席查询 |
| ML 跟踪 | MLflow | 3.10.1 | 实验、参数、指标、模型 artifact |
| 编排 | Dagster | 1.7.x（默认） | Asset/Job/Schedule/Sensor/Asset Check |
| 数据校验 | Pydantic v2 | 2.12.5 | 摄入层逐条记录校验 |
| 数据格式 | Parquet（snappy） | — | Bronze/Silver 默认格式 |
| 日志 | structlog | 25.5.0 | 结构化事件日志 |
| 测试 | pytest | 9.0.2 | 单元 + 集成 |

所有服务都跑在 `docker/docker-compose.yml` + `docker-compose.openmetadata.yml` + `docker-compose.superset.yml` 三份 compose 文件之上，由 `Makefile` 一次拉起（`make up`）。

## 4. 数据流（Medallion 全链路）

```
ECB SDW REST API     DAX 样本 CSV
        │                  │
        ▼                  ▼
   Python collectors（Pydantic 校验、reject 记录）
        │
        ▼
   Bronze Parquet（按 ingestion_date 分区，不可变）
        │
        ▼
   Silver Parquet（类型清洗、去重、衍生字段：rate_change_bps、daily_return）
        │
        ▼
   Gold Parquet（ECB+DAX join，事件研究特征）
        │
        ├── Hive 外表注册：hive.gold.ecb_dax_features
        └── Iceberg Gold（经 Trino CTAS）：iceberg.gold.ecb_dax_features_iceberg
                │
                ├──► Superset 仪表盘
                ├──► OpenMetadata 元数据/血缘
                └──► MLflow（XGBoost/LightGBM + TimeSeriesSplit）
```

存储桶：
- `sololakehouse`：`bronze/`、`silver/`、`gold/`
- `mlflow-artifacts`：MLflow artifact

## 5. 编排层（Dagster v2 已交付）

**Asset 依赖图**：
```
ecb_bronze       dax_bronze
    │                │
ecb_silver       dax_silver
       \            /
        gold_features
              │
         ml_experiment
```

**已交付的运行时形态**：
- `full_pipeline_job`：单条 Dagster job，串起所有 asset
- `daily_pipeline_schedule`：cron `0 6 * * 1-5`（工作日 06:00 UTC）
- `ecb_data_freshness_sensor`：每 30 分钟检查 ECB 新鲜度，必要时触发 `ecb_bronze`
- Asset check：基础新鲜度/形状校验
- 持久化：Dagster 实例存储位于 PostgreSQL `dagster_storage` 库

**入口**：
- `make pipeline` 跑一次 `full_pipeline_job`
- `make dagster-ui` 打开 `http://localhost:3000`

仓库已沉淀 16 份 ADR，分为三组：

| 组 | ADR | 主题 |
|---|---|---|
| 基础选型 | ADR-001 | Docker Compose vs Kubernetes |

|  | ADR-004 | ECB + DAX 数据集选择 |
|  | ADR-005 | v1 范围（为何 Prom/Grafana/CloudBeaver 后置） |
|  | ADR-013 | Gold 用 Iceberg via Trino |
|  | ADR-014 | OpenMetadata（v2.5 起进入默认栈） |
|  | ADR-016 | 计算引擎迁移路径 |
| v2 编排 | ADR-006 | Dagster 编排迁移与回退 |
| v3 治理与生产化 | ADR-007 | K8s + Helm + Terraform 基线 |
|  | ADR-008 | 环境晋升门控（dev → staging → prod） |
|  | ADR-009 | 密钥与访问治理 |
|  | ADR-010 | SLO 驱动的可观测性 |
|  | ADR-011 | ML 生产化边界（先实验，后 serving） |
|  | ADR-012 | 数据治理与目录策略 |
|  | ADR-015 | 可观测性工具选型 |

设计哲学（贯穿 ADR）：
- **本地优先**：可在单机跑通是不可妥协的"可读性约束"
- **开放表格式**：Gold 层用 Iceberg，避免被特定引擎锁死
- **失败即抛**：质量检查抛异常，不做静默降级
- **时间序列禁随机 CV**：ML 一律 TimeSeriesSplit
- **不堆砌组件**：Prometheus/Grafana 不在 v2.5 默认栈，等可观测性叙事成熟再加

## 8. 现状的真实边界（What does not yet exist）

按 `TASKS.md` 当前 bottleneck 列表，v2.5 之上仍然缺失：

| 缺口 | 后果 |
|---|---|
| 多环境模型（dev/staging/prod） | 现在只有"本地默认"一种形态；任何 prod 场景都还需要工程师自己分叉配置 |
| 晋升与回滚纪律 | 没有可执行的 promotion/rollback checklist，只有叙事 |
| 密钥与访问治理 | 凭据走环境变量 + 本地默认；没有统一密钥管理方案落地 |
| SLO 与告警 | 没有可观测性指标管道、没有告警规则、没有事故演练证据 |
| 关键数据集治理契约 | Gold 表的 owner/SLA/quality_class/consumers 未在仓库里以契约形式记录 |
| ML 生产化纪律 | 实验可追溯，但训练输入 → 模型版本 → Gold 数据版本的 lineage 字段未统一 |
| K8s / Helm / Terraform 路径 | ADR-007/008 有方向，但没有 chart、没有 module、没有 values 分层 |

这些缺口正是 v3 计划所要解决的，但 **截至 2026-05-05，它们都还没有动工**。

## 11. 设计哲学（贯穿全仓库的几条隐含规则）

1. **可读性是一等公民**：技术栈刻意保持小到能在一台机器上跑通，但完整到足以讨论生产权衡。
2. **失败要明显**：Pydantic 校验失败、质量检查不达标、asset check 不通过——一律抛错，不做软降级。
3. **元数据要外置**：Hive Metastore 是"表是谁、在哪儿、什么 schema"的唯一真相源，避免把元数据散在脚本里。
4. **Bronze 永不更新**：摄入只追加分区；任何修复都通过新分区 + 新 silver/gold 重算实现。
5. **凭据来自环境**：`os.environ.get()` + 模板（`envsubst`）—— 仓库里不写明文凭据。
6. **时间序列就是时间序列**：禁止随机 CV，禁止往前看的特征，禁止 leak。
7. **每个 ADR 写下"我们没选什么"**：所有架构决策都说出 alternative 与 rejection reason。
