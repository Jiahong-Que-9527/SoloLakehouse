# SoloLakehouse 项目快照

**快照日期：2026-03-26**

## 1. 项目是什么

SoloLakehouse 是一个我正在持续打磨的 Lakehouse 参考实现项目，目标是用一套可运行、可讲解、可验证的代码与部署结构，模拟一个小型现代数据平台从数据采集到分析建模的完整链路。

它不是一个抽象框架，而是一个真实工程化仓库，强调以下几点：

- 能在本地单机环境跑起来
- 能体现 Lakehouse 分层架构
- 能展示数据工程、平台工程、ML 实验管理之间的衔接
- 能通过版本演进清楚说明“我已经做到了哪里、下一步准备做什么”

截至本次快照，项目主线已经从 v1 的线性脚本执行，演进到了 v2 的 Dagster 资产编排模式。

---

## 2. 当前版本与阶段判断

### 当前版本状态

- **v1.0：已交付**
- **v2.0：当前主线**
- **v3.0：规划中**

### 版本含义

- **v1** 重点解决的是：项目能部署、能跑通、能验证，形成一个完整的五层 Lakehouse 核心基线
- **v2** 重点解决的是：项目能编排、能调度、能治理、能可视化回放，把原来的线性 pipeline 升级为资产化平台运行方式
- **v3** 计划解决的是：生产化部署、环境治理、安全控制、可观测性和发布流程

一句话概括当前阶段：

**这个项目已经不是单纯的 ETL Demo，而是一个面向内部 MVP 场景的数据平台参考实现。**

## 3. 当前项目实际包含哪些内容

### 3.1 数据源

当前项目使用两类数据源：

- **ECB API**
  - 用于获取欧央行利率相关数据
- **DAX 日频样例数据**
  - 当前仓库内含示例文件 `data/sample/dax_daily_sample.csv`

这种组合的目的不是做复杂业务，而是用一个足够小、但能串起“宏观事件 + 市场反应 + 特征构建 + ML 实验”的练习场景，来承载平台能力验证。

### 3.2 代码目录结构

当前仓库的核心目录大致如下：

- `ingestion/`
  - 数据采集、schema 校验、Bronze 写入、质量检查
- `transformations/`
  - Bronze -> Silver -> Gold 转换逻辑
- `ml/`
  - 训练、评估、实验记录
- `dagster/`
  - v2 编排层，包括 assets、resources、definitions、io manager
- `scripts/`
  - 运行入口、验证脚本、环境辅助脚本
- `docker/`
  - Compose 文件与各服务 Dockerfile
- `config/`
  - Trino、Postgres 初始化、Hive Metastore 等配置
- `tests/`
  - 单元测试与集成测试
- `docs/`
  - 架构文档、路线图、ADR、部署说明、版本演进说明

### 3.3 当前覆盖的五层链路

项目目前已经覆盖以下五层核心链路：

1. **Source**
   - ECB API
   - DAX CSV
2. **Ingestion**
   - Python collectors
   - Pydantic schema 校验
   - 质量检查
   - rejected records 落盘
3. **Storage**
   - MinIO
   - Parquet
   - Bronze / Silver / Gold
4. **Query & Metadata**
   - Trino
   - Hive Metastore
   - PostgreSQL
5. **ML**
   - 特征数据集
   - 模型实验
   - MLflow 跟踪

## 5. 当前已经落地的核心流程

### 5.1 Legacy 线性 pipeline

项目保留了 v1 时代的线性执行脚本：`scripts/run-pipeline.py`。

该脚本当前仍可运行，主流程是 6 个步骤：

1. ECB Ingestion
2. DAX Ingestion
3. ECB Bronze -> Silver
4. DAX Bronze -> Silver
5. Silver -> Gold
6. ML Experiment

这个入口现在主要承担两个作用：

- 作为兼容路径，保留回退能力
- 作为对比，说明项目是如何从 v1 迁移到 v2 的

### 5.2 Dagster 资产编排主路径

当前 v2 的主路径已经切换到 Dagster，资产定义在 `dagster/assets.py` 中，当前资产包括：

- `ecb_bronze`
- `dax_bronze`
- `ecb_silver`
- `dax_silver`
- `gold_features`
- `ml_experiment`

它们的依赖关系是：

`ecb_bronze` + `dax_bronze`  
-> `ecb_silver` + `dax_silver`  
-> `gold_features`  
-> `ml_experiment`

这意味着项目已经具备了清晰的资产依赖图，而不是只靠脚本中的顺序调用。

### 5.3 调度与治理能力

Dagster 侧已经落地了以下平台能力：

- `full_pipeline_job`
  - 统一执行完整资产链路
- `daily_pipeline_schedule`
  - 工作日 `06:00 UTC` 自动调度
- `ecb_data_freshness_sensor`
  - 每 30 分钟检查一次 ECB Bronze 分区是否过旧
- `gold_features` asset check
  - 检查 Gold 特征数据至少有 10 行，避免下游建模基线过弱

这部分是 v2 和 v1 最大的差别之一，因为它把“运行脚本”升级成了“平台语义上的资产编排”。

当前 `ingestion/` 目录已经实现：

- `ingestion/collectors/ecb_collector.py`
- `ingestion/collectors/dax_collector.py`
- `ingestion/schema/ecb_schema.py`
- `ingestion/schema/dax_schema.py`
- `ingestion/quality/bronze_checks.py`

### 6.2 Medallion 转换层

当前 `transformations/` 目录已经实现：

- `ecb_bronze_to_silver.py`
- `dax_bronze_to_silver.py`
- `silver_to_gold_features.py`
- `quality_report.py`

已落地的能力包括：

- Bronze -> Silver 清洗与字段标准化
- Silver -> Gold 特征构建
- Silver 层质量报告输出
- Gold 层生成面向 ML 的特征数据集

### 6.3 ML 层

当前 `ml/` 目录已经实现：

- `train_ecb_dax_model.py`
- `evaluate.py`

这部分目前的作用主要是：

- 基于 Gold 层特征进行实验

## 7. 当前已实现的平台工程能力

### 7.1 本地部署与启动

当前项目已经支持以下本地命令入口：

- `make setup`
  - 检查 Docker
  - 补 `.env`
  - 拉取镜像
  - 启动服务
- `make up`
  - 启动全部服务并等待健康检查完成
- `make verify`
  - 验证服务健康状态
- `make pipeline`
  - 默认执行 v2 Dagster pipeline
- `make pipeline-v1`
  - 执行 legacy v1 路径

### 7.2 健康检查与可验证性

Compose 中已经为关键服务配置 health check，项目也有单独的 `scripts/verify-setup.py` 来做环境验证。

这意味着项目不只是“启动容器”，而是已经开始考虑：

- 服务是否真的 ready
- 上游依赖是否健康
- 本地首次运行是否容易失败
- 出问题时是否有明确验证入口

### 7.3 查询与注册

当前脚本中还包含 Trino SQL 执行与 Gold 表注册逻辑，说明项目已经开始从“文件级数据处理”走向“表级可查询数据资产”的表达方式。
