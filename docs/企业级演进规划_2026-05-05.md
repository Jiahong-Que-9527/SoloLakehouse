## 0. 总览：五个真实企业问题（North Star）

| # | 企业问题 | 价值定位 | 主要受众 |
|---|---|---|---|
| **P1** | 监管合规的数据血缘追踪 | 最高价值 / 监管必答 | DACH 中型银行、资管、交易所（DekaBank、Commerzbank、Deutsche Börse 类） |

| **P4** | 跨云 / 混合云数据统一层 | 战略架构必备 | 任何在做多云战略的金融机构 |

> 五个问题不是并列；它们之间存在**强依赖**——P1 是其余四个问题的"治理底座"。规划路径据此安排。

## 2. P1 — 监管合规的数据血缘追踪（最优先）

### 2.1 合规上下文

| 法规 / 标准 | 关键条款 | 它要求什么 |
|---|---|---|
| **DORA**（生效 2025-01-17） | Art.6 ICT 风险框架、Art.8 ICT 资产识别 / 业务功能映射、Art.17 事件管理流程 | 关键金融数据资产必须有可识别、可追溯的数据流；监管事件报告窗口不是平台审计证据响应窗口 |
| **BaFin MaRisk AT 4.3.4** | 数据质量与一致性 | 数据治理流程必须有"数据所有者、用途、质量、加工链路"的可证据记录 |
| **MiFID II RTS 22 / RTS 24** | 交易报告字段、记录保存 | 报告字段的 lineage 必须可验证；RTS 24 记录保存通常按至少 5 年口径处理，具体保留期需匹配客户法域与内部政策 |
| **ECB TRIM** | 数据治理目标审查 | 关键模型输入数据的 lineage 必须文档化、可审计 |

### 2.2 当前真实痛点（DACH 现状）

> 中型银行 / 资管的现状：数据散落在 Snowflake、Databricks、本地 Oracle，**血缘靠 Excel 维护**，遇到 BaFin 现场检查靠加班拼接。

### 2.3 SoloLakehouse 已具备的"半成品零件"

- `OpenMetadata` 1.5.x 在默认栈中（v2.5 已交付）→ 资产元数据 + UI
- `Iceberg` Gold 表带原生 `snapshot_id` / `manifest` / `commit_log` → 不可变变更历史
- `Dagster` asset graph + run history → 编排层 lineage（"这次跑用了哪些上游 asset"）
- `Pydantic` schema + reject 落地 → 摄入层证据
- `structlog` 结构化事件日志

零件在，**胶水不在**。

### 2.4 工程动作（v2.6 落地）

**E1. 三源 lineage 融合**
- 建立 `governance/lineage_join.py`，把以下三者按数据集 ID 关联：
  - OpenMetadata 资产元数据（owner、SLA、质量等级）
  - Iceberg snapshot（数据版本、提交时间、修改记录）
  - Dagster run（编排版本、上游 asset、执行人/触发器）
- 输出统一的 `LineageRecord` JSON schema
- 每次 `full_pipeline_job` 跑完后写入 `audit/lineage/<dataset>/<run_id>.json`

**E2. 证据包导出 CLI**
- 实现 `make lineage-evidence DATASET=ecb_dax_features_iceberg DATE=2026-04-30`
- 输出：
  - 数据集快照（schema + snapshot_id + 行数）
  - 上游来源链（直至原始 source）
  - 编排上下文（Dagster run url + 时间戳 + 触发器）
  - 质量证据（asset check 结果、reject 数）
  - 责任人 / 业务用途 / SLA / 质量等级（来自数据契约）
  - PGP 可签名 manifest

**E3. 数据契约文件**
- 新建 `governance/datasets/*.yaml`：
  ```yaml
  dataset_id: ecb_dax_features_iceberg
  data_owner: data-platform@example.eu
  business_purpose: "ECB rate event study features for DAX response model"
  refresh_sla: "T+1 by 06:30 UTC"
  quality_class: "B (model input, non-regulatory-reporting)"
  consumers: ["mlflow:ecb_dax_response_v3", "superset:dashboard:dax_event_study"]
  source_of_truth: "ecb_silver + dax_silver via Dagster gold_features"
  retention: "7 years"
  classification: "internal-confidential"
  ```
- CI 校验（schema + 必填字段）
- 与 OpenMetadata 双向同步（每日 sync job）

**E4. WORM 不可变审计存储**
- 启用 MinIO Object Lock 模式于 `audit/` 桶
- 配置默认保留期：MiFID II RTS 24 场景至少 5 年；如客户法域、合同或内部政策要求更长保留期，通过数据契约覆盖
- Dagster run history 每日外排到 `audit/dagster-runs/<date>.parquet`

**E5. 演练**
- 制造一个"假事件"：BaFin 24h 之内要求提供 `ecb_dax_features_iceberg` 在 2026-04-15 的完整血缘证据
- 跑一次 `make lineage-evidence`，记录用时与产物
- 把 drill 报告归档到 `docs/governance-v3-runbook.md`

### 2.5 对应 TASKS.md
- Block A1（数据集治理基线）
- Block A2（质量门）
- Block H：Lineage Evidence（已加入 `TASKS.md`）

## 3. P2 — 数据主权与厂商锁定逃离

### 3.1 合规上下文

| 法规 | 关注点 |
|---|---|
| **EU Data Act**（2025-09 起适用） | 数据主权与互操作性、可移植性 |
| **Schrems II / GDPR** | 跨大西洋数据传输受限 |
| **DORA Art.28** | ICT 第三方依赖管理、退出策略 |
| **EBA Guidelines on outsourcing arrangements（EBA/GL/2019/02）第 70 段** | 关键服务必须有可执行的退出策略，并能在合理时间内换出 |

### 3.2 关键事实

欧洲企业（尤其 DACH）对 AWS / Azure / Snowflake 的依赖已上升到**董事会议题**。这不是技术问题，是地缘政治问题——**地缘政治问题的预算最大**。

### 3.3 SoloLakehouse 的天然优势

仓库现状已经满足以下条件：
- 100% 开源组件（MIT/Apache 2.0/AGPL，可商用）
- 零美股 SaaS 依赖
- 任意层都可换：MinIO ↔ Ceph ↔ S3，Trino ↔ Spark ↔ Flink，PostgreSQL ↔ 任意 PG 兼容
- Gold 是 Iceberg（开放表格式）—— **这一条是放进合规文档的关键**

### 3.4 工程动作

**E6. 主权证据生成器**
- 实现 `scripts/generate-sovereignty-report.py`
- 输出 `docs/sovereignty-report.md`：
  - 每个组件 → 维护方 → 总部国家 → 许可证 → 是否要求 phone-home
  - 每个外部依赖 → 是否欧盟可替代
  - 每个数据落点 → 物理位置控制方式

**E7. 厂商替代矩阵**
- 在 `docs/portability-matrix.md` 中列出每层组件的：
  - "如果厂商明天消失"的替代方案
  - 替代成本估算（人天）
  - 数据迁移路径（特别是 Iceberg → Spark / Flink / DuckDB 的演示）

**E8. 多引擎兼容性 demo**
- 写一个 `examples/multi-engine/` 目录
- 同一个 Iceberg Gold 表，用 4 种引擎读：
  - Trino（默认）
  - Spark（PySpark）
  - DuckDB
  - Flink（流式读）
- 每个 demo 5 分钟可跑通，附输出截图
- 这是 Iceberg "开放性"的最强证据

**E9. 退出剧本**
- `docs/exit-playbook.md`：如果客户从 AWS / Snowflake 迁出，SoloLakehouse 路径是什么
- 包括：Iceberg metadata 重新指向新 object store、Trino catalog 切换、Helm values 调整

### 3.5 与合规文档的接口

最终交付物应能直接复制粘贴进客户的：
- DORA ICT third-party dependency register
- EBA outsourcing exit plan
- 内部 IT 审计的"开放性合规自评"

## 4. P3 — 中小金融机构的"穷人版 Databricks"

### 4.1 商业现实

- Databricks 对 200 人资管：**€500k+/年**
- 加上 Snowflake / Fivetran / dbt cloud / 数据团队人力：**€1.2M+/年**起
- SoloLakehouse 目标：**1/10 TCO，80% 能力**

### 4.2 SoloLakehouse 现状对比

| 能力域 | Databricks | SoloLakehouse v2.5 | 缺口 |
|---|---|---|---|
| 表格式 | Delta Lake | Iceberg ✓ | — |
| SQL 引擎 | Photon | Trino ✓ | 性能层（Photon 优化未做） |
| 编排 | Workflows | Dagster ✓ | UI 体验差距 |
| ML 平台 | MLflow（Hosted） | MLflow ✓ | 模型 serving 缺位（已在 ADR-011 划界外） |
| 数据目录 | Unity Catalog | OpenMetadata ✓ | 行级权限缺位 |
| 治理 | Unity Catalog policies | 数据契约文件（待建） | 全套缺位 |
| BI | Databricks SQL Dashboards | Superset ✓ | 体验差距 |
| 协作 Notebook | Databricks Notebook | — | 缺位（用 JupyterHub 补） |
| 多用户/团队 | Workspace | — | 缺位（Helm values 未做） |
| 部署 | Multi-cloud SaaS | Compose（v2.5） / Helm（v3.0 计划） | 多节点部署链路 |

### 4.3 工程动作（v2.9 先固化 evidence/readiness，v3.0 再做真实运行时）

**E10. 多节点 Helm chart**
- 实现 `helm/sololakehouse/` chart
- values 分层：`values-dev.yaml`、`values-staging.yaml`、`values-prod.yaml`
- 关键服务支持 HA：Trino coordinator/worker、Hive Metastore、PostgreSQL

**E11. 多用户 / RBAC**
- Trino 接入 LDAP / OIDC（EE 版本无依赖，OSS 配置即可）
- Superset OAuth → 角色映射
- OpenMetadata SSO

**E12. JupyterHub 接入**
- 在默认栈中加入可选的 JupyterHub（profile 控制）
- 预装 Trino client + MLflow client + iceberg-python
- 每个用户独立 namespace

**E13. TCO 计算器**
- `tools/tco-calculator/`：用户输入"用户数、数据量、查询并发"，输出：
  - SoloLakehouse on-prem 估算
  - SoloLakehouse on EU IaaS（Hetzner / OVH / IONOS）估算
  - Databricks 对照估算
- 输出 PDF 报告（适合放进客户提案）

**E14. 一键迁入工具**
- `tools/migrate-from-databricks.py`：扫描 Delta 表 → 转换为 Iceberg → 注册到 Trino
- `tools/migrate-from-snowflake.py`：UNLOAD → Parquet → Iceberg

### 4.4 卖点话术（用于内部 / 客户对话）

> "你不需要换思想——你换的是账单。Lakehouse 的设计语言是 Databricks 教给行业的；SoloLakehouse 让你用同样的语言、付 1/10 的钱，并且把数据主权拿回手里。"

## 5. P4 — 跨云 / 混合云的数据统一层

### 5.1 关键洞察

Iceberg 不是"另一个表格式"，它是**未来 5 年企业数据架构的 lingua franca**。它让以下事情同时成立：
- 同一份数据可以被 Trino、Spark、Flink、DuckDB、Snowflake、BigQuery 读取
- 同一份数据可以横跨 AWS S3、Azure Blob、GCS、MinIO
- 同一份数据可以在不同引擎之间迁移而不改变 schema

### 5.2 SoloLakehouse 的角色

**作为参考实现（不是产品）**，SoloLakehouse 演示：
- Iceberg 表怎么建（`ingestion/trino_sql.py` 的 CTAS）
- Hive Metastore 怎么作为 catalog（也可换 REST catalog）
- 多引擎怎么读同一张表（参见 E8）

### 5.3 工程动作（与 P2 部分共享）

**E15. REST Catalog 替代演示**
- 填实 `docs/decisions/ADR-017-iceberg-rest-catalog-option.md`：Hive Metastore vs Iceberg REST Catalog vs AWS Glue
- 提供切换 demo（compose profile）

**E16. 跨 object store 演示**
- 同一个 Iceberg 表，metadata 指向 MinIO，data 文件分布在 MinIO + 模拟 S3（用第二个 MinIO 实例假扮）
- 演示读穿透多个 storage backend

**E17. Streaming + Batch 统一**
- 加入 Flink 简单 demo：Flink streaming 写入同一张 Iceberg 表
- 演示 Trino batch 读 + Flink stream 写的并发安全（Iceberg ACID）

### 5.4 合规接口

这一条与 **DORA Art.28** 直接对接：客户必须能在合理时间内换出关键 ICT 服务。SoloLakehouse 的多引擎 demo 就是这个"可换出性"的证据。

|---|---|---|
| **EU AI Act** | 2026-08-02 高风险系统义务适用（部分条款有更早 / 更晚时点） | 高风险系统须有：风险管理、数据治理、技术文档、可追溯性、人为监督、稳健性、网络安全 |
| **EU AI Act Art.13** | 同上（高风险系统透明度义务） | 透明度义务：使用者必须能理解系统输出 |
| **EU AI Act Art.10** | 同上（高风险系统数据治理义务） | 训练数据集必须满足质量、相关性、代表性，并保留记录 |
| **ECB Guide to internal models** | 2024 update | 当客户把本平台用于受监管内部模型时，模型输入、假设、变更与验证证据需可审计 |
| **BaFin** | 已生效 | 算法治理：可解释、可审计、可挑战 |

### 6.2 现状

SoloLakehouse 已经具备：
- MLflow 跟踪每次实验（参数、指标、artifact）
- TimeSeriesSplit CV（避免 leak）
- 训练代码进 git，可追溯

但**不够**：
- 模型 run 与 Gold 数据版本（Iceberg snapshot）没有强绑定
- 没有 model card 自动生成
- 没有推理时的 evidence id
- ComplianceRAG / AuditCopilot / LineageNarrator 概念未落地

### 6.3 工程动作

**E18. ML 全链路 lineage 绑定**
- 填实 `docs/decisions/ADR-018-ml-lineage-five-tuple.md`，确认 ML lineage 五元组
- 在 `ml/train_ecb_dax_model.py` 的 `mlflow.start_run()` 上下文里：
  - 自动 log `iceberg.snapshot_id`（Gold 表当前快照）
  - 自动 log `dagster.run_id`（如果在 asset 内执行）
  - 自动 log `feature_version`（来自数据契约）
  - 自动 log `code_commit`
- 这样每个 model run 都能反向查到"它吃了哪份数据"

**E19. 自动模型卡**
- 实现 `ml/generate_model_card.py`
- 训练结束后自动生成符合 EU AI Act Art.13 的 model card：
  - 训练数据来源 / 时间范围 / 行数 / 数据契约引用
  - 训练 / 验证 / 测试切分方式
  - 性能指标（含子群指标）
  - 已知偏差与限制
  - 用途与禁用场景
  - 监控建议
- 写入 MLflow artifact + `audit/models/<model_id>/card.md`

**边界说明：Art.10 evidence form**

v2.8 只提供 EU AI Act Art.10 所需的 evidence form：训练数据来源、切分、质量检查、代表性说明、已知限制和记录保存位置。它**不**自动判断这些内容在客户场景中是否充分、准确或构成合规；内容验证仍需客户的数据治理、模型风险管理和法务流程完成。

**E20. 推理证据 ID（v3.1）**
- 模型 serving 时，每次预测输出附带 `decision_evidence_id`
- 该 ID 可在 audit 系统查询：
  - 用了哪个 model_run
  - 该 run 用了哪个 data_snapshot
  - 该 snapshot 来自哪条 Dagster run
  - 输入特征的实际值
- 这是 **EU AI Act Art.13 透明度义务**的具体落地

**E21. ComplianceRAG（v3.1，独立子系统）**
- 数据源：
  - OpenMetadata 数据资产元数据
  - 仓库内所有 ADR + governance docs
  - Iceberg snapshot 元数据
  - Dagster run history
- 输出：用自然语言回答合规审计问题，每个回答附引用 + 数据版本 + 时间戳
- 例：
  - "Show me the data sources of model X as of 2026-04-30"
  - "Did asset Y meet its SLA in Q1 2026?"
  - "Who owns dataset Z and what's its quality class?"
- 模型用本地可部署的（Llama 3 系 / Mistral / Claude API + 数据不出境的部署形态）

**E22. AuditCopilot（v3.1，独立子系统）**
- 主动型 RAG：扫描运行时事件 → 生成"如果 BaFin 明天来"的预答辩材料
- 每周自动出一份 `audit/weekly-readiness-<date>.md`

**E23. LineageNarrator（v3.1）**
- 把 lineage graph（来自 E1）自动翻译成监管偏好的叙述风格
- 输出符合 BaFin / DORA 报告模板的 PDF

### 6.4 关键边界（与 ADR-011 一致）

- v3.0 仍然 **不做模型 serving**（Triton / vLLM 不在 v3.0 范围内）
- E20–E23 属于 v3.1，前置条件是 v3.0 的治理底座建成
- AI 层永远建在合规底座之上，**而不是绕过合规底座**

## 9. 关键风险与决策点

| 风险 | 影响 | 缓解 |
|---|---|---|
| 监管合规叙事过载，工程交付不足 | 文档漂亮但跑不起来 | 每个能力都必须有可执行的 `make` 入口；不接受"只在文档里"的合规 |
| 跨度过大，v3.0 推迟 | 失去时间窗口（DORA 已生效，AI Act 高风险系统义务在 2026-08-02 适用） | M1 死死锁定 P1，不让 P2–P5 抢资源 |
| ComplianceRAG 显得像玩具 | 失去严肃客户 | 必须做引用追踪 + 数据版本绑定，否则不发布 |
| "穷人版 Databricks" 话术得罪潜在合作伙伴 | 商业关系受损 | 对外措辞改为"on-prem-first lakehouse for sovereignty-conscious institutions" |
| 单人维护难度上升 | 项目停滞 | 严格按 ADR-011 边界，不扩展 serving；AI 层在 v3.1 起作为独立仓库 |

---

## 10. 一句话总结

> **SoloLakehouse 已经有了"小而完整"的开源 Lakehouse 内核；下一步的真正价值，不在于把它做得更大，而在于把它做成 DORA / BaFin / MiFID II / AI Act 的可执行答案——让每一个合规要求都能在仓库里指到具体的 `make` 命令、具体的代码文件、具体的证据产物。这就是 v3.0 要做完的事。**
