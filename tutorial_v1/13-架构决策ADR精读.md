# 13 架构决策 ADR 精读

## ADR-001 Docker Compose vs Kubernetes

- 选择：先 Compose
- 原因：学习曲线低、可本地复现、开发反馈快
- 代价：生产级弹性与多节点能力延后

## ADR-002 Trino vs DuckDB

- 选择：Trino + Hive Metastore
- 原因：更贴近企业数据平台的“元数据 + 查询引擎”范式
- 代价：服务复杂度提高

## ADR-003 Parquet vs Delta

- 选择：Parquet
- 原因：v1 重点在端到端流程与工程模式，不先引入 ACID 复杂度
- 代价：事务与时间旅行能力不足

## ADR-004 数据集选择

- 选择：ECB + DAX
- 原因：公开可访问、无密钥、时间依赖明显、便于构造事件研究特征

## ADR-005 v1 范围控制

- 选择：可观测性和 SQL UI 能力后置
- 原因：先把五服务核心跑稳，再扩展外围能力

## 面试回答技巧

- 不要只说“选了什么”，要说“为什么在这个阶段选它”。
- 强调“阶段性最优”，不是“绝对最优”。
- 给出未来可逆路径（如 v3 上 K8s，或引入 Delta）。
