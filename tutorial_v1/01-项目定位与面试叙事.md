# 01 项目定位与面试叙事

## 项目定位

- **不是框架产品**，是“可运行、可阅读、可演进”的参考实现。
- 核心目标是模拟 Databricks/Snowflake 一类平台的内部层次，而非追求最大吞吐。
- 业务域选择金融时间序列（ECB 利率 + DAX 指数），便于展示时间依赖与数据质量约束。

## 你在面试里的叙事主线

### 1. 背景与问题

“很多人知道 Lakehouse 概念，但很少有能本地一键拉起、覆盖采集到训练全链路、并且带质量门禁和演进路径的参考项目。”

### 2. 我做了什么

- 构建五层核心：数据源、采集验证、Medallion 存储、SQL 查询、ML 跟踪。
- 加入工程化能力：健康检查、失败重试、测试覆盖、CI 门禁。
- 形成发布和演进体系：`v1.0.0` 发布、`docs/history` 追踪决策演化。

### 3. 技术结果

- `make setup -> make verify -> make pipeline` 可复现。
- 质量与类型检查纳入 CI，降低回归风险。
- 可以给出清晰的 v2（编排）和 v3（基础设施）升级路线。

### 4. 商业/团队价值

- 新人可在短时间理解平台架构全貌。
- 方案具备“教学 + PoC + 面试展示”三重价值。
- 为后续团队化建设保留了标准化接口和决策记录。

## 项目边界（必须诚实讲）

- v1 不是生产级高可用集群，定位单机参考实现。
- 可观测性和自助 UI 层在路线图中，未全部在 v1 完成。
- 数据规模与复杂度有限，重点在工程模式而非极限性能。

## 30 秒版本话术

“SoloLakehouse 是我做的 Lakehouse 参考平台工程项目：单机 Compose 拉起 MinIO、Postgres、Hive Metastore、Trino、MLflow，数据从 ECB/DAX 进入 Bronze/Silver/Gold，再跑 ML 实验。它不是框架，而是可运行的端到端工程样板，重点展示平台工程里的可靠性、质量门禁和架构演进能力。”

## 2 分钟版本话术

“这个项目主要解决两个问题：第一，Lakehouse 概念落地难，很多案例停留在图；第二，面试里很难证明自己不仅懂组件，还懂工程化。  
我用五层核心把链路打通：采集层做了 schema 校验和 rejected records；转换层按 Medallion 生成可消费数据；查询层用 Trino+Hive Metastore；ML 层用 MLflow 管实验。然后我把可靠性和可验证性补齐：服务健康检查、pipeline 重试策略、CI 的 lint/typecheck/test/coverage。  
在路线图上，v2 计划引入 Dagster 做资产级编排，v3 再上 K8s/Terraform。我还加了 history 文档体系，记录每个版本为什么这么选，保证演进可追溯。”
