# 14 v1 到 v4 演进路线与取舍

## 路线图摘要

- **v1**：核心闭环 + 易部署 + 可验证
- **v2（当前）**：Dagster 编排、调度、资产视图
- **v2.5（参考）**：Gold Iceberg + 可选 OpenMetadata（教学/演示，不替代 v3 生产清单）
- **v3（规划）**：Kubernetes/Helm/Terraform 基础设施化
- **v4（规划）**：自助化成熟与运维体验强化

## 演进原则

1. 先稳定再复杂：先脚本稳定，再引入编排器。
2. 先可复现再扩规模：先单机可复现，再云原生化。
3. 每次升级都保留回退策略：避免一次性重构导致不可控风险。

## v2 关键取舍

- Dagster 带来资产可视化与调度，但增加运行时复杂度。
- 需保留 `pipeline-legacy` 作为迁移缓冲。

## v3 关键取舍

- K8s/Terraform 提升可扩展与环境一致性。
- 同时增加运维复杂度、成本与团队门槛。

## 面试表达模板

“我不是一次把所有先进组件都堆上去，而是按风险递进：v1 先把链路跑稳并形成质量门禁，v2 再解决编排与运维可视化，v3 才进入基础设施生产化。这种路线对团队更可执行，也更容易验证每一步价值。”
