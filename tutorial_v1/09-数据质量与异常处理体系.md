# 09 数据质量与异常处理体系

## 质量检查分层

### Bronze 检查（强约束，失败即中断）

- `check_no_nulls`
- `check_no_future_dates`
- `check_date_continuity`
- `check_schema_version`

### Silver/Gold 报告（软约束，记录不阻断）

- `run_silver_quality_report`：输出行数、空值、日期范围、重复。

## rejected records 机制

- 校验失败记录统一写入 `bronze/rejected/source=...`。
- 每条附带 `rejection_reason`。
- 提升可追溯性与数据治理能力。

## 异常分类

- **输入不可用**：如上游 API 超时，归类 `CollectorUnavailableError`。
- **输入不合法**：schema 校验失败，进 rejected。
- **处理异常**：pipeline 统一封装为 `StepError`，带 step metadata。

## 面试表达

“我把错误分成可重试的外部依赖问题、可隔离的数据问题、必须终止的处理问题。这样一方面保证数据主链路纯净，另一方面不丢失错误样本，便于后续修复和审计。”
