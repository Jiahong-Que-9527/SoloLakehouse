# 07 观测性与故障排查实战

## 核心结论

面试官真正关心的是：系统挂了你怎么定位。v2 的排障要从“服务状态 + Dagster run + 结构化日志 + 数据产物”四个面同时看。

---

## 诊断四板斧

1. **服务面**：`make verify` / `docker compose ps`
2. **编排面**：Dagster UI 的 run timeline、asset events、checks
3. **日志面**：`pipeline_metric`、失败事件、异常栈
4. **数据面**：MinIO 路径是否生成、行数/质量是否达标

---

## 典型故障场景清单

### 场景1：作业未按时触发

- 查 `dagster-daemon` 是否运行
- 查 schedule 是否启用、时区是否正确
- 查 sensor 是否一直返回 `SkipReason`

### 场景2：Bronze 成功、Gold 失败

- 定位 `gold_features` 资产日志
- 验证上游 Silver parquet 是否完整
- 检查 schema 变化或数据窗口不足

### 场景3：ML 资产失败

- 检查 Gold 数据行数（asset check）
- 检查 MLflow endpoint 与 artifact 存储可用性
- 检查参数网格导致的训练异常

---

## 日志与指标怎么讲

v2 增加了结构化 step 时延指标：

- `metric = pipeline.step.duration_ms`
- 可用于横向比较各资产耗时变化

面试官若追问“为什么不是 Prometheus”：

答：当前阶段先保证结构化可查询日志，降低系统复杂度；在 v3/生产化再接指标系统和告警链路。

---

## 故障排查 SOP（可背）

1. 先判断是 **服务不可用** 还是 **作业逻辑失败**
2. 若服务问题，先修基础依赖（Postgres/MinIO）
3. 若作业问题，看 Dagster run 的失败资产和异常
4. 验证该资产输入数据是否符合预期
5. 选择资产级重跑而不是全量重跑
6. 记录根因与修复动作，沉淀到文档

---

## 面试话术（2分钟）

“我把排障分成服务层、编排层、日志层、数据层。  
先看依赖服务是否健康，再看 Dagster run 失败资产，再结合结构化指标和数据产物验证根因。  
定位后优先资产级重跑，避免全量回放。  
这个方法的核心是缩短 MTTR，同时减少对下游的影响范围。”
