# 08 发布流程与 GitHub 操作复盘（v2.0.0）

## 核心结论

发布不是“打个 tag”这么简单，而是“代码、文档、历史、版本元数据”一致性的最后验证。

---

## v2 发布闭环应包含的动作

1. 代码层：v2 功能全部落地（含可选项的决策）
2. 文档层：`CLAUDE.md`、`docs/architecture.md`、`docs/DAGSTER_GUIDE.md`、`docs/history/*`
3. 计划层：`docs/EVOLVING_PLAN.md` 勾选完成项
4. 版本层：commit + tag + push + GitHub Release

---

## 你这次已经做过的关键发布操作（可面试复盘）

- 提交了 v2 编排相关核心文件
- 创建并推送 `v2.0.0` tag
- 使用 `gh release create` 生成 release 页面和说明
- 明确未将本地训练资料目录纳入发布内容

---

## 为什么要强调“发布文档一致性”

面试官常见关注点：

- 你是否只会写功能代码，不会做发布治理？
- 版本历史是否能让新同事快速理解“发生了什么”？

正确回答方向：

“我把发布当成工程治理动作，不仅提交代码，还同步 roadmap/history/运行文档，避免版本知识断层。”

---

## 标准发布检查清单（可复用）

- [ ] 工作区只有预期变更
- [ ] 关键命令可跑（up/verify/pipeline）
- [ ] 关键文档同步完成
- [ ] commit message 说明“为什么”
- [ ] tag 命名符合语义版本
- [ ] release notes 可被外部读者理解

---

## 面试题：如果发布后发现严重问题怎么办？

建议回答：

1. 立即冻结新增变更
2. 使用 legacy 路径回退运行（若 Dagster 主路径异常）
3. 快速定位最小修复集
4. 补丁发布并记录 RCA

这能体现你有“可回滚优先”的工程意识。
