# 05 数据层与ML层在v2下的协作

## 核心结论

v2 没有推翻 v1 的数据与 ML 代码，而是用 Dagster 把现有模块“资产化编排”。这是一种低风险、可回滚的演进策略。

---

## 数据流回顾（在 v2 仍成立）

`ECB API / DAX CSV -> Bronze -> Silver -> Gold -> MLflow`

变化点：

- v1：`scripts/run-pipeline.py` 线性调用
- v2：Dagster 资产依赖图驱动调用

---

## 资产与原模块映射表（面试必背）

- `ecb_bronze` -> `ECBCollector.collect()`
- `dax_bronze` -> `DAXCollector.collect()`
- `ecb_silver` -> `ecb_bronze_to_silver.run()`
- `dax_silver` -> `dax_bronze_to_silver.run()`
- `gold_features` -> `silver_to_gold_features.run()`
- `ml_experiment` -> `run_experiment_set()`

---

## 为什么这种“包装式迁移”有价值

1. **复用成熟逻辑**：减少回归风险
2. **降低迁移成本**：不需要同时改业务+编排+存储
3. **快速获得平台能力**：调度、重跑、lineage、资产检查

---

## ML层在 v2 的关键面试点

- 仍通过 MLflow 记录实验
- 训练仍基于 Gold 特征
- v2 的主要贡献不是“模型精度提升”，而是“模型流水线运营能力提升”

这类回答能显著加分，因为你在表达正确的工程优先级。

---

## 如何回答“你做了哪些平台化改造”

可直接复述：

“我没有先动业务特征工程，而是先把数据与 ML 过程接入统一编排语义。  
这样我们先得到可运行历史、可重跑、可调度、可检查，再逐步迭代模型策略。  
这符合平台演进的路径：先控流程，再优算法。”

---

## 进一步可扩展点（系统设计追问）

- 增加 model registry 推进上线流程
