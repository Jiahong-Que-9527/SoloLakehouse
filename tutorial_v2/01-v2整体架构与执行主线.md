# 01 v2整体架构与执行主线

## 核心结论

v2 将 SoloLakehouse 从“线性脚本 orchestrator”升级为“Dagster 资产编排平台”，并通过 Docker Compose 维持本地低门槛部署。

---

## v2 的平台结构（你要会画）

```text
User/CLI
  └─ make pipeline
      └─ dagster-webserver (job execute)
          ├─ 资产执行（bronze/silver/gold/ml）
          ├─ 元数据/运行事件
          └─ 与 dagster-daemon 协作（schedule/sensor）

Runtime Services
  - minio
  - postgres
  - hive-metastore
  - trino
  - mlflow
  - dagster-webserver
  - dagster-daemon
```

---

## 从 `make pipeline` 到模型实验：端到端主线

1. `Makefile` 默认 `pipeline` 执行 Dagster job  
2. Dagster 载入 `dagster/workspace.yaml -> definitions.py -> defs`
3. `full_pipeline_job` 按资产依赖执行：
   - `ecb_bronze` / `dax_bronze`
   - `ecb_silver` / `dax_silver`
   - `gold_features`
   - `ml_experiment`
4. 每个资产内部调用已有 v1 模块（collectors/transformations/ml）
5. 运行元数据和事件写入 Dagster 存储（PostgreSQL）
6. 在 UI 可查看 run、asset lineage、schedule、sensor、checks

---

## 为什么这是平台工程而不是“换个工具”

- **抽象层提升**：从“步骤”变成“资产”（数据产物为中心）
- **恢复能力提升**：失败后可按资产重跑，而不是全量脚本重跑
- **治理能力提升**：内建 schedule、sensor、asset check
- **可观测性提升**：统一运行历史 + 结构化指标

## 面试高频追问与回答要点

### 问：为什么不直接删掉 legacy script？

答：迁移期保留双路径是为了控制变更风险。Dagster 主路径稳定后可逐步收敛到单路径，避免一次切换导致线上/演示不可回滚。

### 问：v2 是否已经“生产级”？

答：v2 解决了编排和可运营性，但基础设施仍是单机 Compose。真正生产级会在 v3 引入 K8s/Helm/Terraform 与环境级治理。
